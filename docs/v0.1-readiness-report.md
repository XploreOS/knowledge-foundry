# Knowledge Foundry v0.1 — Readiness Report

Date: 2026-07-05 · Orchestrator: Fable 5 · Status: **v0.1 complete — all 18 acceptance criteria pass**

## 1. What was built

- **Claude Code skillset** (`skill/`): `SKILL.md`, `CLAUDE.md` (binding operating contract), `AGENTS.md`, 12 slash-command definitions, 11 agent definitions with prompts and guardrails, 4 hook contracts mapping to deterministic gate functions.
- **TypeScript monorepo** (npm workspaces, ESM, Node ≥ 20, strict tsc):
  - `@knowledge-foundry/core` — 13 zod schemas + enums (the frozen data contract), storage module owning the entire artefact layout, pure gate functions (`preIngestGate`, `citationGate`, `licenseConsistencyGate`, `preReleaseGate`), and one module per pipeline stage (registry, licensing, ingestion, normalization, chunking, tagging, claims, risk, conflicts, release, evals).
  - `@knowledge-foundry/cli` — the `kf` CLI: 15 commands, commander-based, every input/output schema-validated, gate refusals print explicit `BLOCKED` banners with non-zero exits.
  - `@knowledge-foundry/adapters` — localFs, web (opt-in fetch), markdown, JSON, CSV; PDF is an explicit v0.1 stub that throws with guidance.
  - `@knowledge-foundry/domain-templates` — generic, healthcare, legal, finance, enterprise (7 schema-valid YAML files each).
- **Functional medicine reference domain** (`domains/functional-medicine/` + `examples/functional-medicine/`) — 9 entity types, A–X evidence model, 6 risk rules (warfarin interactions, UL exceedance, hormone supervision, PHI, off-label, license), legal/clinical_sme/cmo/product_owner review workflow, 8 eval questions. Fully deletable; the engine contains zero domain-specific strings.
- **Documentation** (`docs/`): getting-started, architecture, domain-config, license-policy, release-model, marketplace-readiness, functional-medicine-reference-domain, decisions (ADR-001…014), implementation-log, and `docs/internal/contract-spec.md` (the normative cross-package contract).
- **Test suite** (`tests/`): 60 Vitest tests in 10 files — unit coverage of every gate and schema plus three CLI integration suites (full happy-path pipeline, Red-source blocking, missing-citation blocking) in isolated temp workspaces.

## 2. What works (verified)

- `npm install`, `npm run build`, `npm test` all succeed (fresh run 2026-07-05: **10 files, 60/60 tests, ~24s**).
- Full pipeline end-to-end via CLI: init-domain → validate-domain → create-source → classify-license → ingest → normalize → chunk → tag → extract-claims → screen-risk → detect-conflicts → build-release → validate-release → eval-rag (all exit 0 on a toy Green corpus; eval metrics pass the generic template thresholds).
- **Safety gates, enforced in code and covered by tests:**
  - Red sources: terminally `rejected` at classification, blocked at ingest (no raw artefact written), blocked at release.
  - Yellow/Orange: blocked from ingestion until a human approval is recorded (`classify-license --approve`).
  - Unknown/unclassified: blocked from ingestion.
  - Chunks with empty citations: block the release with the offending chunk id named.
  - Unresolved high-severity risk flags: block the release.
  - `fine_tuning` intended use: blocked unless every member source has explicit training rights; classification never invents rights.
- Raw artefacts are write-once with SHA-256 checksums, verified before normalization.
- Releases are directories with machine-readable `manifest.json` (counts, license class counts, evidence summary, gate results, blockers, member-file checksums, embedded evaluation).

## 4. Intentionally deferred (out of v0.1 scope, per master prompt)

- Vector database / RAG application integration (releases are RAG-ready JSONL; indexing is the consumer's job).
- Review UI — review decisions are recorded through CLI state changes.
- PDF extraction (explicit stub with actionable error).
- Web-search-based discovery — `kf discover` imports agent/human-produced JSONL candidate lists (no paid APIs in basic mode).
- Zip/tarball release export; database backends; hosted SaaS.
- LLM-powered tagging/claim extraction — core ships deterministic scaffolds; the skill's agents perform semantic enrichment and the CLI validates/persists it.

## 5. Known limitations

- Tagging, claim extraction, and conflict detection heuristics are deliberately simple (keyword/numeric-threshold based); they are correctness-safe scaffolds meant to be augmented by Claude agents through the skillset.
- `evalRag` uses term-overlap retrieval — a proxy metric, not an embedding-based retriever.
- Yellow/Orange approval is recorded via `classify-license --approve` by a human operator; there is no multi-reviewer quorum enforcement in the CLI yet (declared in `review_workflow.yaml`, enforced organizationally in v0.1).
- Windows line endings are normalized via `.gitattributes` (`eol=lf`); contributors on Windows should keep autocrlf off or trust the attributes file.

## 6. How to run the CLI

```bash
npm install && npm run build
npm link -w @knowledge-foundry/cli   # exposes `kf`
kf --help
kf init-domain demo && kf validate-domain demo
```

Workspace root defaults to cwd; override with `--root <dir>` or `KF_ROOT`. Full walkthrough: `docs/getting-started.md`.

## 7. How to use the Claude Code skill

Load the `skill/` directory as a Claude Code skill bundle. `skill/SKILL.md` declares scope and non-negotiable rules; `skill/CLAUDE.md` is the operating contract (agents orchestrate, `kf` gates enforce); `skill/commands/*.md` map 1:1 to `kf` commands; `skill/agents/*.md` contain per-stage prompt templates and escalation rules; `skill/hooks/*.md` define the four gate checkpoints.

## 8. How to extend with new domains

`kf init-domain <id> --template <generic|healthcare|legal|finance|enterprise>`, edit the 7 YAML files, `kf validate-domain <id>`. Schema shapes: `docs/domain-config.md` and `docs/internal/contract-spec.md` §3. No engine changes needed — ADR-013.
